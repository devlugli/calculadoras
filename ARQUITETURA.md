# Filosofia de Arquitetura do CalcBrazil

O CalcBrazil não deve ser apenas um conjunto de páginas soltas. A proposta do projeto é ser um portal de utilidades online com navegação clara, expansão organizada e foco em pageviews, recorrência e monetização.

## Princípio central

Cada página precisa ter um papel claro dentro do portal e levar o usuário naturalmente para a próxima.

Em resumo:

`menos páginas soltas, mais ecossistema de navegação`

## 1. O site é um portal de utilidades

O projeto não deve ser pensado apenas como um site de calculadoras.

Ele está crescendo para incluir:

- calculadoras
- ferramentas de texto
- geradores de dados
- guias

Toda nova página precisa nascer dentro de um sistema claro:

- tipo de conteúdo
- categoria
- página-hub
- páginas relacionadas
- possível guia complementar

Regra:

nenhuma página deve parecer isolada.

## 2. A home distribui, não inventaria

A home não precisa mostrar tudo. A função dela é:

- apresentar o portal
- destacar páginas fortes
- enviar tráfego para hubs
- gerar segunda e terceira pageview

Regra:

a home vende descoberta, não inventário completo.

## 3. Hubs são a base da arquitetura

Páginas agregadoras como:

- `calculadoras-financeiras`
- `calculadoras-datas`
- `calculadoras-combustivel`
- `calculadoras-matematica`
- `calculadoras-saude`
- `guias`

devem funcionar como hubs de navegação, SEO e descoberta.

Regra:

sempre que um grupo crescer, ele deve ganhar um hub próprio.

## 4. Toda página deve empurrar para outra

A navegação ideal do portal não termina no primeiro clique.

Cada página deve abrir caminho para continuidade:

- calculadora → relacionadas
- calculadora → populares
- calculadora → categoria
- guia → ferramenta relacionada
- hub → páginas individuais
- home → hubs e páginas fortes

Regra:

o usuário nunca deve parar no primeiro clique.

## 5. Navegação global curta, navegação contextual forte

Não é ideal encher o topo ou a lateral com dezenas de links logo cedo.

O caminho preferido do projeto é:

- menu principal enxuto
- hubs bem estruturados
- links contextuais dentro das páginas

Regra:

menos poluição global, mais contexto local.

## 6. Crescimento organizado por intenção

O site precisa organizar o conteúdo de acordo com o que o usuário quer fazer.

Exemplos de intenção:

- calcular
- gerar
- consultar
- escrever
- aprender

Regra:

o usuário deve entender rapidamente qual tipo de ferramenta resolve o problema dele.

## 7. Monetização vem de navegação melhor

Mais pageviews e melhor monetização tendem a vir de:

- boa distribuição interna
- hubs fortes
- páginas relacionadas
- conteúdos complementares
- descoberta fácil

Não vêm apenas de:

- menu enorme
- excesso de links
- páginas duplicadas
- navegação confusa

Regra:

mais páginas por sessão, não mais confusão por sessão.

## 8. Todo novo bloco precisa nascer pronto para escalar

Quando entrarem novas áreas como ferramentas de texto ou geradores, elas já devem nascer com:

- slug consistente
- hub próprio
- padrão visual
- padrão de textos
- links internos previsíveis

Regra:

o site não deve crescer no improviso.

## 9. O visual comunica função

O design deve ajudar o usuário a entender o papel de cada bloco.

Exemplos:

- card de ferramenta direta
- card de categoria
- card de guia
- card de novidade

Regra:

hierarquia visual também é arquitetura.

## 10. O site precisa parecer vivo

Para gerar recorrência, confiança e crescimento, o portal precisa mostrar evolução constante.

Isso pode aparecer em:

- novas ferramentas
- novos guias
- hubs atualizados
- blocos de destaque
- melhorias contínuas na home

Regra:

catálogo vivo gera retorno; catálogo parado gera visita única.

## Regra final

Antes de criar qualquer nova página, seção ou menu, vale fazer esta pergunta:

`isso ajuda o usuário a encontrar algo, usar algo e continuar navegando?`

Se a resposta for não, provavelmente não fortalece a arquitetura do portal.
